# `v0.7.0` Accuracy Improvements

This note covers the retrieval quality improvements from `v0.4.0` to `v0.7.0`.

## Benchmark Summary

All versions measured with the same local-binary benchmark harness, pinned repositories, and local Jina CUDA ONNX stack.

- 21 tasks
- 4 repos: `ripgrep`, `flask`, `fastify`, `turborepo`
- hybrid retrieval with BM25, dense embeddings, and reranking

## What Changed

### Richer indexed text

Earlier versions fed raw chunk content to both BM25 and embeddings. The current pipeline prepends structured metadata (language, file path, filename, symbol name, signature) to each chunk before indexing. This gives both lexical and semantic search stronger signal for config lookups, symbol disambiguation, and path-based queries.

### BM25 field boosting

The BM25 index now stores filename and path tokens as separate searchable fields with configurable boost weights. Path-weighted queries (e.g. "Cargo.toml workspace configuration") boost filename matches, while symbol queries boost the symbol_name field. This replaced the previous flat-text BM25 approach.

### Direct symbol and filename lookup

For identifier queries (e.g. "Config", "Blueprint class"), the search pipeline performs a direct SQLite lookup by symbol name alongside the normal retrieval path. Case-sensitive matches are preferred, and exact type definitions rank ahead of incidental method name matches. Filename lookups work the same way for file-specific queries.

### Single-pass query-aware ranking

A lightweight heuristic reranking pass runs once after the final retrieval stage. It adjusts scores based on file role (source, test, docs, config), path depth, symbol type match, and filename relevance. Earlier versions applied this heuristic three times at different pipeline stages, which caused the heuristics to dominate over neural signals. The single-pass design keeps the neural retrieval pipeline's ranking as the primary signal.

### Cleaner evaluation

The benchmark harness fixes a bug where overlapping chunks could claim credit for the same ground-truth entry, inflating nDCG. Duplicate hits are now de-duplicated during scoring, and raw ranked results are saved for inspection.

### Reranker always active

The cross-encoder reranker now runs on all query types. An earlier version skipped reranking for path-weighted queries, silently degrading to heuristic-only ranking for an entire query class.

## What This Means

From `v0.4.0` to `v0.7.0`, the pipeline improved by:

- `+0.4762` Recall@1
- `+0.2738` Recall@5
- `+0.3095` Recall@10
- `+0.4079` MRR@10
- `+0.3791` nDCG@10

The older public comparison tables in the README compare Vera against other tools on the same workload and remain valid as a cross-tool reference. The `v0.7.0` local benchmark numbers reflect the current state of the retrieval pipeline.
